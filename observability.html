Silent Failures & Data Drift

At 1.6B events/month, "silent failures" (APIs changing, latency creeping up) were our biggest churn risk. Customers would find out before we did.

I led the implementation of a comprehensive Grafana & Datadog monitoring layer. We moved from "ticket-driven development" to proactively catching latency spikes (like the 17hr backlog shown here) before customers churned.

Anomaly Detection & Triage

Dashboards are great, but no one watches them 24/7. I defined the SQL logic for our Anomaly Audit system.

If a pipeline stalls for >2 hours, an alert fires. I often jump into the read-replica DB to triage root cause myself before paging engineering.

SELECT COUNT(*), DATE(time_created), DATE_PART('hour', time_created) FROM arnie_enterprise_prod.public.entity WHERE entity_type = 'Appointment' GROUP BY 2, 3 ORDER BY 2 DESC, 3 DESC;

Example: A triage query I use to identify hourly ingestion gaps or spikes during an active incident.

Global averages hide problems. I pushed for multi-tenant observability so we could see if a single massive customer was clogging the pipe.

This granularity allows us to route traffic dynamically and prevent one heavy user from taking down the platform for everyone else.

The "Weekly Ritual" (Culture Change)

Tools don't fix reliability; culture does. I established the Weekly Reliability Review ritual with Engineering Leadership.

  • Review: We reviewed every alert and "near-miss" from the previous week.
  • Action: If an alert fired too often (noise), we tuned it. If a silent failure occurred, we added a monitor.
  • Result: This feedback loop is what actually drove the 63% reduction in tickets—we stopped fixing the same bug twice.
