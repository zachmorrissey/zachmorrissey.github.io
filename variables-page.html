Variable Exploration UI

A "Data X-Ray" for the platform. I designed this interface to solve the "Black Box" problem, giving non-technical users total transparency into their schema, data quality (fill rates), and field usage without needing to ping Engineering.

Design PhilosophyDensity

Function Over Form

Customers often have messy data (e.g., three different fields named "Address"). They didn't need a pretty card view; they needed a high-density diagnostic tool.

I designed this grid to show maximum rows per page, with a Global Search that cuts across all entities (Candidate, Job, Placement). This prevents users from checking the wrong entity silo and missing critical data.

The "Data Command Center." Note the cross-entity search and "Fill Rate" indicators.
Data Quality

The "Fill Rate" Metric

The biggest question users had was: "Is this field actually being used?"

I introduced a background job that calculates the Non-Empty Value % (Fill Rate) for every field. If a user sees "Address 1 (5%)" and "Address 2 (95%)," they instantly know which variable to map, preventing thousands of failed messages downstream.

The Inspector Panel: Shows Top 10 values, unique value counts, and where this variable is used.
Practical AIExplainability

Translating Python to English

We have "Derived Fields" (e.g., date_last_note) that are calculated via complex Python logic. Customers (and CS) often forgot what triggered them.

I implemented an LLM-powered feature that reads the backend Python config and generates a human-readable tooltip definition on the fly. This turns "config code" into clear, actionable logic.

Contextual AI: Hovering over a derived field reveals the underlying logic in plain English.
Troubleshooting

The "Test Record" Debugger

Sometimes you just need to check one specific person. I added a Test Record mode that lets users search for a known entity (e.g., "James Vince") and see exactly what values Sense has stored for them.
